## Ce qui a bien fonctionné

### 1. Architecture "handlers-only" pour Story 2.2
La décision de ne pas modifier les composants enfants et de router tous les 7 checks via les handlers existants de MoteurView a été la meilleure décision de l'Epic. Cela a :
- Évité 7 modifications de composants brownfield
- Centralisé toute la logique dans un seul fichier
- Simplifié considérablement les tests (harness de handlers au lieu de mount de composants)

### 2. Pattern de spy Pinia découvert en Story 2.2
Le problème "Pinia actions are not spies" a été résolu avec `vi.spyOn(progressStore, 'addCheck').mockResolvedValue()`. Ce pattern est devenu le standard pour toutes les stories suivantes (réutilisé en Story 2.3).

### 3. Composants légers et ciblés
ProgressDots.vue (97 lignes) et PhaseTransitionBanner.vue (~85 lignes) sont des composants simples avec des responsabilités claires. Pas de sur-ingénierie.

### 4. Continuité du mapping PHASE_CHECKS
Le mapping checks→phases défini dans ProgressDots (PHASE_GROUPS) a été réutilisé dans MoteurView pour la logique de transition. Cohérence maintenue.

### 5. Tests par responsabilité
La structure de tests par responsabilité (helper logic, handler integration, backend duplicate, component rendering, banner visibility) a facilité la lisibilité et la maintenance.

## Défis rencontrés

### 1. `@pinia/testing` non installé (Story 2.1)
Premier test tenté avec `createTestingPinia` — module non disponible. Résolu en utilisant `setActivePinia(createPinia())` + manipulation directe du state.

**Leçon :** Ne pas assumer la présence de packages optionnels. Vérifier `package.json` avant d'utiliser un import.

### 2. Pinia actions non espionnées (Story 2.2)
`expect(progressStore.addCheck).toHaveBeenCalledWith(...)` échouait avec "is not a spy". Les actions Pinia créées avec `createPinia()` sont des fonctions réelles, pas des vi spies.

**Leçon :** Toujours utiliser `vi.spyOn()` sur les actions Pinia avant de faire des assertions `toHaveBeenCalled`.

### 3. ExplorationVerdict handler inline (Story 2.2)
Le template avait `@continue="activeTab = 'audit'"` comme expression inline. Il a fallu extraire vers une fonction nommée `handleExplorationContinue` pour y ajouter `emitCheckCompleted`.

**Leçon :** Les inline handlers dans les templates rendent l'ajout de logique plus difficile. Préférer les fonctions nommées.

## Insights clés

1. **"No child modification" pattern** : Quand un wrapper (MoteurView) intercepte déjà les événements des enfants, il est plus propre et moins risqué d'ajouter la logique dans le wrapper plutôt que de modifier chaque enfant.

2. **Computed chains pour les bandeaux** : La chaîne `currentPhaseId → isCurrentPhaseComplete → transitionBanner → showTransitionBanner` est un pattern élégant pour contrôler l'affichage conditionnel réactif. Chaque computed a une responsabilité unique.

3. **Watch pour reset de state UI** : Le `bannerDismissed` reset via watch sur `currentPhaseId` et `selectedArticle.slug` montre que les watches sont appropriés pour les effets de bord UI, pas les computed.

4. **Backend déjà prêt** : L'article-progress service avait déjà la protection contre les doublons (`includes` check). L'infrastructure backend existante a accéléré l'Epic.
