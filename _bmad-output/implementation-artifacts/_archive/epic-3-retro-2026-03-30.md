## Ce qui a bien fonctionné

### 1. Architecture progressive à travers les 3 stories
L'Epic a suivi une progression naturelle : Story 3.1 a posé l'infrastructure (route backend, type StrategyContextData, store fetchContext, composant collapsable), Story 3.2 l'a exploitée côté backend (enrichissement prompts), et Story 3.3 côté frontend (scoring alignement). Chaque story a construit sur la précédente sans nécessiter de refactoring.

### 2. Réutilisation des types et services existants
L'interface `StrategyContextData` créée en Story 3.1 a été réutilisée directement en Story 3.3 pour le scoring d'alignement. Le service `cocoon-strategy.service.ts` est resté inchangé — aucune modification du backend de données stratégiques. Le principe "lecture seule sur l'existant" a parfaitement fonctionné.

### 3. Pure functions pour la logique métier
Le choix de `computeAlignmentScore` comme pure function (pas un composable avec state) a simplifié les tests considérablement : pas besoin de monter un composant ou initialiser Pinia, tests unitaires purs avec `describe/it/expect` uniquement.

### 4. Normalisation NFD cohérente
Le même pattern `normalize('NFD').replace(/[\u0300-\u036f]/g, '')` est utilisé dans : slugify (backend), prompt-loader, useAlignmentScore. La cohérence évite les bugs de matching accentué.

### 5. Backward-compatible signature changes
L'extension de `loadPrompt()` avec un 3ème paramètre optionnel `options?: { cocoonSlug?: string }` n'a cassé aucun caller existant. Même pattern pour `generateRadarKeywords()` avec `cocoonSlug?`. Zéro régression.

## Défis rencontrés

### 1. Full-keyword containment vs token matching (Story 3.3)
Le test `useAlignmentScore` pour "renovation energetique" échouait initialement : le test attendait `matchedTerms` contenant le token individuel `'renovation'`, mais la fonction retourne `'renovation energetique'` (full keyword match, score 80). La distinction entre les deux branches (full keyword containment → score 80 avec kwNorm, token matching → score ratio) n'était pas anticipée dans les assertions.

**Leçon :** Quand une fonction a plusieurs branches de matching avec des retours différents, écrire les tests en pensant à quelle branche sera empruntée pour chaque input.

### 2. Résolution cocoonId → cocoonSlug (Story 3.1)
Les stratégies sont stockées par slug (`data/strategies/cocoon-{slug}.json`) mais le frontend connaît le cocoonId (numérique). Il a fallu résoudre via `getCocoons()` pour trouver le nom puis slugifier. Ce mapping indirect ajoute un point de fragilité.

**Leçon :** Quand deux systèmes utilisent des identifiants différents (id numérique vs slug string), documenter la résolution dans le code et tester les cas edge (cocoon inexistant, nom avec caractères spéciaux).

### 3. Dual enrichment coexistence (Story 3.2)
`buildStrategyContext()` (article-level, dans generate.routes.ts) et `buildCocoonStrategyBlock()` (cocoon-level, dans prompt-loader.ts) coexistent. La confusion potentielle entre les deux a nécessité des commentaires explicites et une séparation claire des responsabilités.

**Leçon :** Quand deux fonctions ont des noms similaires et des rôles proches mais distincts (article vs cocoon), nommer explicitement pour lever l'ambiguïté et documenter la coexistence.

## Insights clés

1. **Pattern "infra → backend → frontend"** : L'ordre Story 3.1 (infra + UI) → 3.2 (backend enrichissement) → 3.3 (frontend scoring) a minimisé les allers-retours. Chaque story avait un scope clair et des dépendances unidirectionnelles.

2. **Computed Map pour memoization** : Dans KeywordAuditTable, le `alignmentMap = computed<Map<string, AlignmentResult>>()` évite de recalculer `computeAlignmentScore` à chaque render/sort. Pattern réutilisable pour toute colonne calculée dans un tableau.

3. **Conditional column pattern** : `v-if="props.strategicContext"` sur `<th>` et `<td>` est le pattern propre pour les colonnes optionnelles dans un tableau. Pas besoin de `colspan` ou de logique complexe.

4. **Tests sans Pinia** : L'extraction de la logique métier en pure function (useAlignmentScore) a permis des tests ultra-simples. Pas de `createTestingPinia`, pas de mock, pas de `vi.spyOn`. Juste des inputs et outputs.
