## Ce qui a bien fonctionné

### 1. Découpage clean en 2 stories séquentielles
Story 5.1 (squelette LaboView + route + navigation) était indépendante de Story 5.2 (dual-mode). Ce découpage a permis :
- 5.1 : livraison rapide sans toucher aux composants existants
- 5.2 : focus pur sur le pattern dual-mode avec la fondation déjà en place
- Zéro interférence entre les deux stories

### 2. Analyse pré-implémentation des 10 composants (Story 5.2)
Avant de toucher au code, une exploration exhaustive a identifié que seuls **2 sur 10** composants nécessitaient de la logique conditionnelle. Les 8 autres étaient déjà keyword-only — la prop `mode` est un contrat d'interface, pas une obligation de code conditionnel. Cette analyse a évité 8 `if (mode === 'libre')` inutiles.

### 3. Pattern `withDefaults(defineProps<{...}>(), { mode: 'workflow' })` uniforme
Un seul pattern pour les 10 composants — aucune variation. Le default `'workflow'` garantit la non-régression : tout code existant qui ne passe pas `mode` continue de fonctionner identiquement.

### 4. LaboView comme orchestrateur léger
277 lignes, 4 onglets plats, pas de logique métier — juste du wiring entre keyword et composants. La vue respecte le principe d'un orchestrateur minimal.

### 5. Tests couvrant les deux modes
Les 12 tests dual-mode vérifient que chaque composant accepte `mode='libre'` ET `mode='workflow'` sans crash. C'est du test de contrat, pas du test d'implémentation — exactement ce qu'il faut pour un refactoring prop-level.

## Défis rencontrés

### 1. Mock factory complexe pour useKeywordDiscoveryTab
Le composable `useKeywordDiscoveryTab` expose ~35 propriétés réactives. Le mock factory dans `dual-mode-props.test.ts` fait ~50 lignes. C'est le composable le plus volumineux du projet, et chaque nouveau test qui importe un composant l'utilisant doit reproduire ce mock.

**Leçon :** Quand un composable grossit au-delà de 30 exports, envisager un fichier de mock partagé (`__mocks__/useKeywordDiscoveryTab.ts`) pour éviter la duplication entre fichiers de tests.

### 2. Mock manquant `autoReactivate` (PainValidation test)
Le premier run des tests dual-mode a révélé que `useNlpAnalysis` expose une fonction `autoReactivate()` qui n'était pas dans le mock initial. PainValidation l'appelle dans son setup. Fix rapide (ajout au mock) mais rappelle l'importance de vérifier les imports réels des composants avant d'écrire les mocks.

**Leçon :** Pour les tests de montage, parcourir le `<script setup>` du composant pour lister TOUS les composables appelés et leurs fonctions utilisées.

### 3. Edit tool "file not read yet" (récurrent — 3ème epic consécutif)
Le tool d'édition a bloqué sur PainTranslator.vue, sprint-status.yaml, et KeywordDiscoveryTab.vue (2ème édition). Solution systématique appliquée : Read → Edit, chaque fois.

**Leçon :** Ce pattern est maintenant ancré dans le workflow. Systématiquement relire avant toute édition, même si le fichier a été lu récemment dans la même session.

## Insights clés

1. **Le dual-mode est un problème d'interface, pas d'implémentation.** 80% du travail (8/10 composants) se résume à ajouter une prop optionnelle avec default. Seuls les composants avec des watchers article-dépendants ou du contexte discovery ont besoin de logique conditionnelle. Identifier cette distinction avant de coder a épargné de la complexité inutile.

2. **Audit exclu du Labo — bonne décision.** L'Audit (`KeywordAuditTable`) est scoped par cocoon via `auditStore.fetchAudit(cocoonName)`. Plutôt que de forcer un mode libre artificiel, l'onglet a été simplement exclu du Labo (4 onglets au lieu de 8). Accepter les limites du design existant plutôt que de sur-ingénierer une compatibilité douteuse.

3. **Les 3 tests pré-existants en échec sont une dette chronique.** Ils apparaissent dans chaque retro depuis Epic 2 (navigation-restructuring, production-phases, translate-pain). Ils ne sont pas bloquants mais polluent les rapports et masquent potentiellement de vrais problèmes. Ils méritent un nettoyage dédié hors epic.

4. **L'incrément de 1 test file pass par epic est constant.** Epic 3: 88, Epic 4: 88→89 (correction d'un comptage), Epic 5: 89→90. Chaque epic ajoute 1-2 fichiers de tests. Le projet a maintenant 90 fichiers de tests passants et 1222+ tests unitaires.
