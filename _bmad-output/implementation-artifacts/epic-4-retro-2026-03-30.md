## Ce qui a bien fonctionné

### 1. Utilitaire cache générique bien dimensionné
La fonction `getOrFetch<T>(cacheDir, key, ttlMs, fetcher)` avec son type `CacheEntry<T>` a couvert 100% des besoins. Aucun service migré n'a nécessité d'extension ou de paramètre supplémentaire. Les helpers low-level `readCached`/`writeCached` ont aussi servi pour les services avec une logique cache conditionnelle (autocomplete TTL dynamique, intent writeJson sélectif).

### 2. Migration incrémentale sans casse
Les 8 services ont été migrés un par un vers le nouveau format `{ data: T, cachedAt: string }`. Les anciens fichiers cache sont traités comme cache miss (format incompatible → relecture échoue → re-fetch). Aucun script de migration de données n'a été nécessaire. Zéro régression.

### 3. Story 4.2 — architecture "read-only backend" propre
L'endpoint `GET /articles/:slug/cached-results` ne fait QUE lire les caches existants via `readCached()` — il n'appelle jamais d'API externe. Cette séparation stricte (écriture = services individuels, lecture = endpoint de collecte) est élégante et testable.

### 4. Race condition guard élégant
Le composable `useArticleResults` utilise un simple `currentSlug` ref pour ignorer les résultats stale quand l'utilisateur change d'article rapidement. Pattern minimal et efficace, testé unitairement avec un scénario de double-fetch.

### 5. Tests d'abord pour le cache utility
Les 15 tests de `cache.test.ts` ont été écrits en même temps que l'implémentation. Cela a permis de détecter des edge cases : TTL=0 (toujours stale), fichier corrompu (graceful miss), clé vide.

## Défis rencontrés

### 1. Cascade de mises à jour de tests (Story 4.1)
Le changement de format cache (données brutes → `{ data, cachedAt }`) a nécessité la mise à jour de 6 fichiers de tests existants. Chaque `mockReadJson.mockResolvedValue(cached)` devait devenir `mockReadJson.mockResolvedValue({ data: cached, cachedAt: '...' })`. Le premier run de tests a révélé 9 échecs, dont certains dans des fichiers non anticipés (local-seo.service.test.ts découvert au 2ème run).

**Leçon :** Quand on change un format de données partagé, faire un `grep` exhaustif sur les patterns de mock avant de commencer les corrections. Ne pas se fier au premier run de tests — il peut masquer des fichiers impactés.

### 2. Keyword resolution slug → capitaine (Story 4.2)
Le backend doit résoudre un slug d'article vers son mot-clé principal pour chercher les caches. Deux sources possibles : `SelectedArticle.keyword` (frontend) ou `ArticleKeywords.capitaine` (backend via data.service). Le choix de `getArticleKeywords(slug).capitaine` côté backend est plus fiable (source de vérité) mais ajoute une dépendance à `data/article-keywords.json`.

**Leçon :** Toujours utiliser la source de vérité backend pour les résolutions de données, même si le frontend dispose déjà de l'info. Le frontend peut avoir des données stale.

### 3. Edit tool "file not read yet" (récurrent)
Le tool d'édition a bloqué plusieurs fois avec "File has not been read yet" quand on essayait d'éditer sprint-status.yaml ou server/index.ts après un long intervalle sans les relire. Solution systématique : toujours relire avant d'éditer.

**Leçon :** Pattern workflow : Read → Edit, jamais Edit seul sur un fichier lu il y a longtemps.

## Insights clés

1. **CacheEntry<T> comme contrat universel** : Le wrapper `{ data: T, cachedAt: string }` est assez simple pour convenir à tous les services. La tentation d'ajouter `ttl`, `version`, `source` dans l'entrée cache a été évitée — la simplicité prime.

2. **readCached + writeCached pour le contrôle manuel** : Tous les services n'ont pas pu utiliser `getOrFetch` directement. L'autocomplete a un TTL dynamique (24h normal, 30min vide), l'intent écrit conditionnellement. Les helpers low-level (`readCached`/`writeCached`) comblent ce gap sans complexifier `getOrFetch`.

3. **Endpoint de collecte vs hydratation store-by-store** : Un seul endpoint `/cached-results` qui retourne les 5 types de cache en parallèle est plus efficace que 5 appels API séparés. Le coût réseau est divisé par 5 côté frontend.

4. **Discovery store : reset only** : Le keyword-discovery store n'a pas de cache backend 1:1 par keyword (les résultats dépendent du contexte — seed, domain, etc.). Il est simplement vidé au changement d'article, pas rechargé. Accepter cette asymétrie est plus sain que forcer un cache artificiel.
