## Ce qui a bien fonctionné

### 1. Story 1.1 comme fondation solide
La Story 1.1 (Layout 3 phases) a réalisé l'essentiel de la restructuration en une seule story. Cela a permis aux stories suivantes d'être légères et rapides. Story 1.3 n'a même nécessité aucune modification de code — uniquement des tests de validation.

### 2. Code review efficace
La code review de Story 1.1 a identifié 5 issues concrètes :
- **HIGH** : `recommendedContentLength` mort après retrait de ContentGapPanel — nettoyé
- **MEDIUM** : Cast `as Tab` non sécurisé — remplacé par un runtime type guard avec `TAB_IDS`
- **MEDIUM** : Task 3.3 sur-déclarée comme faite
- **LOW** : Absence d'attributs ARIA — ajouté `role="tablist"`, `role="tab"`, `aria-selected`
- **LOW** : Icône lock emoji — documenté comme acceptable

### 3. Architecture brownfield respectée
Tous les composants internes existants (KeywordDiscoveryTab, DouleurIntentScanner, PainTranslator, PainValidation, ExplorationInput, KeywordAuditTable, etc.) ont été préservés intacts. Seul le wrapper MoteurView et la navigation ont changé.

### 4. Cross-tab communication maintenue
Les 5 patterns de communication inter-onglets (Discovery→Radar, Radar→Validation, PainTranslator→Validation, Validation→Exploration, Exploration→Audit) ont été préservés sans régression.

### 5. Validation build systématique
L'ajout d'une étape `vue-tsc --noEmit` + `vite build` après chaque dev-story a permis de détecter immédiatement tout problème de typage ou de compilation.

## Défis rencontrés

### 1. Complexité de la Story 1.1
La restructuration de MoteurView (passage de tabs plats à 3 phases) était la tâche la plus lourde. Le fichier MoteurView est massif avec de nombreux imports, refs, computed, et handlers. Le risque de régression était élevé.

**Mitigation :** Les tests unitaires du composant MoteurPhaseNavigation ont été isolés du reste, permettant de valider la navigation indépendamment.

### 2. Dead code non détecté avant code review
`recommendedContentLength` était devenu orphelin après le retrait de ContentGapPanel. Ce type de dead code est difficile à détecter automatiquement dans un projet Vue avec `<script setup>`.

**Leçon :** Lors du retrait d'un composant, vérifier systématiquement les refs/computed qui lui étaient liés.

### 3. Tests pre-existants en échec
8 tests dans 2 fichiers (`navigation-restructuring.test.ts` et `production-phases.test.ts`) échouaient avant le début de l'Epic. Cela a créé du bruit dans les rapports de test.

**Impact :** Mineur — ces tests concernent des composants non liés (SiloCard, BrainPhase).

---

## Insights clés

1. **Front-loading works** : Concentrer le gros de l'implémentation dans la première story (1.1) puis faire des stories de validation/complétion (1.2-1.4) produit un flux efficace.

2. **Code review comme filet de sécurité** : La review a rattrapé un bug de type safety (cast non sécurisé) qui aurait pu causer des erreurs runtime silencieuses.

3. **Gating souple = bonne UX** : Le pattern de "message informatif sans blocage" utilisé dans les Stories 1.2 (lock banner) et 1.4 (AssignmentGate) respecte la liberté de navigation tout en guidant l'utilisateur.

4. **Tests isolés par phase** : Structurer les tests par phase (moteur-phase-generer.test.ts, moteur-phase-valider.test.ts, moteur-phase-assigner.test.ts) facilite la maintenance et la lisibilité.
